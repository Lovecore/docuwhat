## Category Configuration

### Using _meta.json

Each folder can have a `_meta.json` file to configure the category:

<CodeBlock language="json">
{
  "title": "Getting Started",
  "description": "Begin your journey with our platform",
  "order": 1,
  "icon": "🚀"
}
</CodeBlock>

### Meta Fields

| Field | Purpose | Example |
|-------|---------|---------|
| `title` | Display name in navigation | `"API Reference"` |
| `description` | Category overview | `"Technical API documentation"` |
| `order` | Sort position | `1`, `2`, `10` |
| `icon` | Visual identifier | `"📚"`, `"🔧"`, `"⚡"` |

## Navigation Hierarchy

### Three-Level Structure

Optimal depth for most documentation:

1. **Category** (folder with _meta.json)
2. **Section** (subfolder or file)
3. **Page** (individual MDX file)

<Callout type="info" title="Depth Recommendation">
  Avoid going deeper than 3 levels. If you need more organization, consider restructuring into separate top-level categories.
</Callout>

### Example Navigation

```
Documentation (Level 0)
├── Getting Started (Level 1)
│   ├── Installation (Level 2)
│   └── Configuration (Level 2)
├── Guides (Level 1)
│   ├── Basic (Level 2)
│   │   ├── First App (Level 3)
│   │   └── Data Fetching (Level 3)
│   └── Advanced (Level 2)
│       ├── Performance (Level 3)
│       └── Security (Level 3)
└── Reference (Level 1)
    ├── API (Level 2)
    └── CLI (Level 2)
```

## Naming Conventions

### Files and Folders

Use consistent naming:

✅ **Good:**
- `getting-started.mdx`
- `api-reference.mdx`
- `troubleshooting-guide.mdx`

❌ **Avoid:**
- `GettingStarted.mdx`
- `getting_started.mdx`
- `gettingstarted.mdx`

### URL-Friendly Names

File names become URLs:
- `user-authentication.mdx` → `/user-authentication`
- `guides/deployment.mdx` → `/guides/deployment`

## Search Optimization

### Effective Tagging

Use tags strategically:

<CodeBlock language="yaml">
tags: ["authentication", "security", "jwt", "oauth"]
</CodeBlock>

**Tag Guidelines:**
- 3-5 tags per page
- Use consistent tag names
- Include category tags
- Add technology tags
- Consider user intent

### Search-Friendly Titles

Write titles that users search for:

✅ **Good:**
- "How to Deploy to Vercel"
- "Authentication with JWT Tokens"
- "Fixing CORS Errors"

❌ **Less Effective:**
- "Deployment"
- "Auth"
- "CORS"

## Maintenance

### Content Review Schedule

Regular maintenance keeps docs fresh:

1. **Weekly**: Check for broken links
2. **Monthly**: Review popular pages
3. **Quarterly**: Update examples and screenshots
4. **Yearly**: Full content audit

### Version Management

For versioned documentation:

```
content/
├── v1/
│   └── (version 1 docs)
├── v2/
│   └── (version 2 docs)
└── latest -> v2/
```

### Deprecation Strategy

Mark deprecated content clearly:

<CodeBlock language="mdx">
<Callout type="warning" title="Deprecated">
  This feature is deprecated. Use [new feature](/new-feature) instead.
</Callout>
</CodeBlock>

## Scaling Tips

### Large Documentation Sets

For 100+ pages:

1. **Use clear categories** - Maximum 7-9 top-level categories
2. **Implement search** - Search becomes critical at scale
3. **Add breadcrumbs** - Help users navigate
4. **Create index pages** - Category landing pages
5. **Use cross-references** - Link related content

### Team Collaboration

When multiple authors contribute:

1. **Style guide** - Document writing standards
2. **Templates** - Provide page templates
3. **Review process** - Establish review workflow
4. **Ownership** - Assign category owners
5. **Changelog** - Track significant changes
